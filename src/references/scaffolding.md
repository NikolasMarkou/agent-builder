# Scenario Scaffolding Reference

Pre-composed pattern recipes for common agent scenarios. Each scenario specifies the topology, behavioral, and data flow patterns to compose (defined in `patterns.md`), a scenario-specific state shape, production guardrails, and failure modes.

Use this reference at Step 2 (pattern selection) when the task matches a known scenario. Use it at Step 4 (build) for state shape templates and scenario-specific guardrails. For pattern definitions, see `patterns.md`. For production hardening details, see `production.md`.

## How to Use This Reference

1. Match your use case to a scenario below.
2. Start with the **baseline topology**. Upgrade only when a measured failure demands it.
3. Copy the **state shape** as your starting TypedDict/Pydantic model.
4. Apply the **guardrails** to your graph before deployment.
5. Review the **failure modes** -- these are expected production behaviors, not edge cases.

If your use case does not match any scenario, fall back to the pattern selection decision framework in `patterns.md`.

## Composition Escalation Rule

When composing patterns, start simple and escalate only when a specific, measured failure demands it:

```
1. Single agent with tools (ReAct)           <- try this first
2. Sequential pipeline                       <- add when steps have strict ordering
3. Sequential + Loop (quality gates)         <- add when quality requires iteration
4. Router + Sequential                       <- add when inputs vary widely
5. Parallel + Aggregator                     <- add when independent tasks exist
6. Hierarchical                              <- add when scale requires delegation
7. Hierarchical + Swarm or Network           <- only when evidence demands it
```

> **Design axiom: Tiered escalation.** Each level adds coordination overhead. A Hierarchical + Swarm system that could have been a Sequential pipeline wastes tokens on supervision. Measure the failure that motivates the upgrade.

## Deep Research

### What Distinguishes It

A RAG pipeline retrieves context and generates once. A deep research agent operates in a **discovery loop**: it searches, synthesizes, identifies what it still does not know, and decides autonomously whether to continue or report. The architectural primitive is **gap-awareness** -- modelling its own ignorance, not just what it has found.

| Property | Shallow RAG | Deep Research Agent |
|---|---|---|
| Query type | Single, predetermined | Evolving, generated by agent |
| Iteration count | 1 | 3-N (bounded) |
| Termination condition | Fixed | Condition-based (gaps empty OR budget exhausted) |
| Knowledge state | Stateless per call | Accumulated across iterations |
| Source handling | Retrieve and forget | Tracked, attributed, deduplicated |
| Planning | None | Explicit decomposition before execution |

### Topology

**Baseline -- Single-Agent Loop (try first):**
```
[Query] -> [Agent: ReAct Loop] -> {search, read, synthesize, check gaps} -> [Report]
                |_______________ [gaps found] __________________________|
```

**Scale -- Hierarchical + Parallel (when single-agent hits context limits):**
```
[Goal]
  |
[Planner/Supervisor]
  | decomposes into N sub-questions
  |--[Specialist A: Domain 1 STORM Loop]
  |--[Specialist B: Domain 2 STORM Loop]
  +--[Specialist C: Domain 3 STORM Loop]
        | parallel execution
  [Aggregator: Synthesis]
        |
  [Report -> Generator-Critic -> HITL gate -> Delivery]
```

**When to use:** Multi-hop questions requiring iterative discovery, literature surveys, competitive analysis, investigative workflows.
**When NOT to use:** Simple fact-lookup (single ReAct call suffices), questions with a single authoritative source, time-constrained queries where iterative search is too slow.

**Adaptive Router (cost-critical):**
```
[Query]
  |
[Complexity Classifier]
  |-- Simple factoid  -> Single LLM call (no agent)
  |-- Moderate        -> Standard RAG pipeline
  +-- Complex multi-hop -> Full STORM loop with specialists
```

> **Design axiom: Tiered escalation.** Routing 60-80% of queries to cheaper paths cuts cost 40-60% vs. running full agentic research on every query.

### Behavioral Pattern: STORM Details

The STORM pattern is defined in `patterns.md` Section 2.5. Scenario-specific rules for deep research:

- **Multi-perspective query generation is mandatory.** Generate queries from at least 3 angles: overview, technical detail, counterargument, recent developments, expert opinion. Single-angle queries miss large portions of the knowledge space.
- **Gap detection is the quality driver.** Prompt explicitly: "What claims are unsupported? What perspectives are missing?" If `len(gaps[n]) >= len(gaps[n-1])`, gap detection is diverging -- halt and escalate.
- **Source tracking is non-negotiable.** Every source is `{url, query, content, timestamp}`. Facts trace to source indices. Without this, synthesis is indistinguishable from hallucination.
- **Iteration limit is both cost control and quality signal.** If gaps persist after 5 iterations, either the question is unanswerable from available sources or query generation is cycling.

### State Shape

```python
class ResearchState(TypedDict):
    topic: str                                                    # immutable after init
    scope: str                                                    # immutable after init
    queries_issued: Annotated[list[dict], operator.add]           # {query, iteration}
    sources: Annotated[list[dict], operator.add]                  # {url, query, content, timestamp, iteration}
    outline: list[str]                                            # updated each iteration
    gaps: list[str]                                               # updated each iteration
    claims: Annotated[list[dict], operator.add]                   # {claim, source_indices}
    iteration: int
    status: Literal["researching", "writing", "reviewing", "done", "failed"]
    draft_report: str
    critique_history: Annotated[list[dict], operator.add]         # {iteration, issues, score}
    final_report: str
```

### Guardrails

| Guardrail | Rule |
|---|---|
| Iteration cap | Hard limit at 5. At 10+ without convergence, log and escalate. |
| Retry budget | Per-query max: 3. After 3 failures, continue without it, log the failure. |
| Source deduplication | Hash URL + first 200 chars. Skip duplicates. |
| Parallel query cap | Max 5 concurrent. Diminishing returns beyond this. |
| Gap convergence check | If `len(gaps[n]) >= len(gaps[n-1])`, halt and escalate. |
| Cost circuit breaker | Estimate tokens before expensive synthesis. If over budget, compact first. |
| HITL before final output | For legal, financial, medical research -- interrupt before delivery. |

### Failure Modes

| Failure | Root Cause | Fix |
|---|---|---|
| Hallucinated citations | Synthesis invents attributions | Store `{url, content}` pairs; cite by index, not by generating URLs |
| Infinite research loop | Gap detection too sensitive | Score gap criticality; continue only on high-severity gaps |
| Context overflow | Verbatim source text accumulates | Compact after each iteration; store summaries, not raw text (`production.md` Context Engineering) |
| Query cycling | Semantically identical queries repeated | Hash queries; skip if similarity to prior query > threshold |
| Early termination | Optimistic synthesizer empties gaps prematurely | Use adversarial gap-finding prompt; require minimum iterations |

## Customer Support / Triage

### What Distinguishes It

High-volume, heterogeneous inputs. Quality is measured in resolution rate and escalation rate, not content quality. The dominant pattern is Router + Specialist, not a loop. Latency matters because it is user-facing.

### Topology

**Baseline -- Router + Specialists:**
```
[User Message]
  |
[Intent Classifier] <- small, fast model
  |-- billing   -> [Billing Specialist Agent]
  |-- technical -> [Tech Support Agent]
  |-- refund    -> [Refund Agent + HITL approval gate]
  |-- complaint -> [Escalation Agent]
  +-- general   -> [FAQ Agent: RAG over knowledge base]
        |
  [Response + Ticket Update]
```

For classifier schema design, confidence thresholding, and routing implementation, see `structured-classification.md`.

**Tier escalation within each specialist:**
```
[Specialist Agent: ReAct Loop]
  |-- resolved                    -> Close ticket, log resolution
  |-- needs approval (high-value) -> interrupt() -> human agent -> resume
  +-- cannot resolve              -> Escalate to human queue with context summary
```

**When to use:** Customer support, helpdesk, ticketing systems, any high-volume intake with distinct request categories.
**When NOT to use:** Tasks where all inputs require the same handling (skip the router), low-volume workflows where a single agent with all tools suffices.

### Behavioral Rules

- **Use the smallest model for classification.** A 7B local model or Claude Haiku handles most intent classification. Using a flagship model here is wasteful -- this node runs on every request. See `structured-classification.md` for model tiering.
- **Always define a fallback route** (`general` or `unknown`). Unclassifiable inputs without a fallback produce silent routing failures.
- **HITL placement:** Trigger interrupt before any write operation with monetary consequence (refund, credit, subscription change). Trigger on low confidence. Never block low-stakes reads (order status, FAQ) with approval gates.

### State Shape

```python
class SupportState(TypedDict):
    ticket_id: str
    user_id: str
    conversation_history: Annotated[list[dict], operator.add]     # {role, content}
    intent: str
    intent_confidence: float
    route: str
    resolution_attempts: Annotated[list[dict], operator.add]      # {action, result}
    escalation_reason: str | None
    status: Literal["open", "in-progress", "resolved", "escalated", "pending-approval"]
    thread_id: str                                                # for durable cross-session state
```

### Guardrails

| Guardrail | Rule |
|---|---|
| Per-ticket tool call cap | Max 10 tool calls per conversation turn. Prevents runaway tool loops. |
| Approval gate scope | Define a fixed set of `HIGH_RISK_TOOLS` (write operations). Gate only those. |
| Session persistence | Use thread-level checkpointing. Agents can resume after human intervention without losing context. |
| Confidence floor | If intent confidence < 0.6, default to general agent rather than risk misrouting. See `structured-classification.md`. |
| Resolution timeout | If unresolved after N turns, surface to human queue automatically. |

### Failure Modes

| Failure | Root Cause | Fix |
|---|---|---|
| Wrong specialist assigned | Misclassification on ambiguous intents | Confidence thresholding; fallback on low-confidence routing |
| Loop on unresolvable issue | No escalation condition | Explicit max-turns-before-escalate in state |
| Context lost on resume | No checkpointing | Thread-level checkpointer (Postgres-backed in production, see `deployment.md`) |
| Approval gate blocking fast-path | All actions gated | Narrow gate scope to explicitly defined high-risk tools only |

## Code Generation & Review

### What Distinguishes It

Output is executable. Correctness is verifiable by running tests, not by LLM judgment. The quality signal is objective (tests pass or fail), which enables tight feedback loops unavailable in prose generation tasks.

### Topology

**Code generation -- Sequential + Loop:**
```
[Spec / Issue / PR Description]
  |
[Planner: decompose into subtasks with dependency graph]
  |
[Coder Agent: ReAct Loop]
  |-- write code
  |-- run linter
  |-- run tests
  +-- interpret results -> loop if failing
  |
[Reflection pass: security, style, edge cases]
  |
[HITL: human review before merge]
```

**Code review -- Parallel + Aggregator:**
```
[PR Diff]
  |
[Parallel review agents]
  |-- [Security Reviewer: OWASP checks, injection, auth]
  |-- [Logic Reviewer: correctness, edge cases, test coverage]
  +-- [Style Reviewer: conventions, maintainability]
        |
  [Aggregator: merge review comments]
  |
[HITL: human approval before posting]
```

**When to use:** Automated code generation, PR review bots, migration assistants, code refactoring at scale.
**When NOT to use:** Simple code completion (single LLM call suffices), code review where a single reviewer covers all dimensions.

### Behavioral Rules

- **Plan-and-Execute for generation.** The planner outputs a dependency graph before any code is written. This prevents writing code that violates downstream constraints. Re-planning triggers when a test failure reveals a wrong assumption.
- **Generator-Critic with executable feedback.** The critic for code is a test runner, not an LLM rubric. Loop exit condition is `tests_pass == True`. When tests are absent, an LLM critic checks: security vulnerabilities, boundary conditions, type correctness, semantic alignment with the spec.
- **Reflexion (cross-task memory).** After a coding session, store critique as memory: "When implementing authentication, I incorrectly assumed JWT expiry was handled by the library." Inject into future sessions with similar tasks.

### State Shape

```python
class CodeState(TypedDict):
    task_description: str
    language: str
    plan: list[dict]                                               # {subtask, dependencies, status}
    code_files: dict[str, str]                                     # filename -> content (latest)
    test_results: Annotated[list[dict], operator.add]              # {run_id, passed, failed, errors}
    lint_results: Annotated[list[dict], operator.add]              # {run_id, issues}
    review_comments: Annotated[list[dict], operator.add]           # {reviewer, severity, file, line, comment}
    revision_count: int
    reflexion_memory: list[str]                                    # lessons from prior sessions
    status: Literal["planning", "coding", "testing", "reviewing", "approved", "failed"]
```

### Guardrails

| Guardrail | Rule |
|---|---|
| Sandboxed execution | Run all generated code in an isolated environment (Docker, E2B). Never execute directly on host. |
| Test coverage gate | Do not accept completion if test coverage < N% on new code. |
| Revision cap | Hard limit on revision cycles (e.g., 5). After cap, surface to human rather than loop further. |
| File write scope | Agent can only write to its designated working directory. Enforce at the tool level, not the prompt level. |
| Dependency pinning | Any new package import must be pre-approved or logged. Prevents supply chain risk. |
| Cost per task budget | SWE agents can cost $5-8 per task unconstrained. Set per-task token budgets explicitly. |

### Failure Modes

| Failure | Root Cause | Fix |
|---|---|---|
| Test satisfaction without correctness | Agent modifies tests instead of code | Read-only tool for test files; only code files are writable |
| Infinite test loop | Flaky tests with non-deterministic output | Detect flaky tests (same code, different results); skip and flag |
| Scope creep | Coder adds unrequested complexity | Spec adherence check in Reflection pass |
| Unsafe code generation | No security review step | Add security-focused Reflection pass before HITL gate |

## Data Analysis & Reporting

### What Distinguishes It

Two knowledge sources must be combined: structured (SQL / dataframes) and unstructured (docs, policies, reports). Neither alone is sufficient. The planner must decompose questions into queries appropriate to each source type.

### Topology

**Parallel (SQL + Search) -> Aggregator:**
```
[Question]
  |
[Planner: decompose into SQL tasks + search tasks]
  |
  |-- [SQL Agent] <- parallel
  +-- [Search Agent] <- parallel
        | (fan-in)
  [Synthesizer: combine structured + unstructured results]
  |
[Quality Gate: confidence >= threshold?]
  |-- Yes -> deliver
  |-- No + retries remaining -> retry with refined plan
  +-- No + retries exhausted -> HITL review
```

**When to use:** Business intelligence, financial analysis, compliance reporting, any task combining database queries with document search.
**When NOT to use:** Pure SQL queries (no agent needed -- generate and execute), pure document search (use RAG instead).

### Behavioral Rules

- **Controlled flow for SQL.** SQL execution is deterministic -- do not run it inside an LLM reasoning loop. The pattern is: generate SQL (LLM step), validate and execute (deterministic tool step), interpret results (LLM step). Collapsing them causes the LLM to "invent" SQL results when the query returns empty.
- **Schema-first planning.** The planner must have the database schema before decomposing the question. Inject schema as structured system context, not as a tool the planner calls mid-execution. This halves the time to first useful query.
- **Confidence-gated synthesis.** The synthesizer returns a confidence score reflecting data completeness, not output fluency. Low confidence triggers retry or HITL gate.

### State Shape

```python
class AnalysisState(TypedDict):
    question: str
    plan: dict                                                     # {sql_tasks, search_tasks, reasoning}
    sql_results: Annotated[list[dict], operator.add]               # {task, query, result} -- accumulator
    search_results: Annotated[list[dict], operator.add]            # {task, result} -- accumulator
    synthesis: str
    confidence: float
    caveats: list[str]
    sources: list[str]
    iteration: int
    status: Literal["planning", "executing", "synthesizing", "complete", "review"]
```

**Accumulator typing rule:** Both `sql_results` and `search_results` use `Annotated[list, operator.add]` reducers so parallel branches do not overwrite each other. This is the most common implementation bug in parallel data agents.

### Guardrails

| Guardrail | Rule |
|---|---|
| Read-only SQL enforcement | Enforce `SELECT` only at the tool level. Reject any non-SELECT statement before execution. |
| Row cap | Cap SQL results at 50-100 rows returned to the agent. Full results available via aggregation queries. |
| Schema injection | Always inject schema as system context, not as a runtime tool call. |
| Confidence threshold | Default: 0.7. Below this, do not deliver results without human review. |
| Retry limit | Max 2 retries on low-confidence synthesis. After that, surface to human with context. |

### Failure Modes

| Failure | Root Cause | Fix |
|---|---|---|
| SQL hallucination | LLM "fills in" missing query results | Strict separation: LLM generates SQL; deterministic executor runs it |
| Schema confusion | Agent uses wrong table/column names | Schema injected at planning time; validation node before execution |
| Silent empty result | Empty SQL result treated as "no data" rather than "query may be wrong" | Log empty results explicitly; trigger query review on empty return |
| Overconfident synthesis | LLM synthesizer inflates confidence | Require caveats list; tie confidence to result completeness, not output fluency |

## Document Processing Pipelines

### What Distinguishes It

High throughput, heterogeneous input types, strict output schemas. Success is measured by extraction accuracy and processing throughput, not reasoning quality. The dominant pattern is Router + Type-Specific Subgraph, not a reasoning loop.

### Topology

**Router -> Type-Specific Subgraphs:**
```
[Document Intake]
  |
[Parser: extract text, tables, images]  <- reusable subgraph
  |
[Type Classifier: contract | invoice | report | form | other]
  |-- contract -> [Contract Subgraph: extract clauses -> risk flag -> legal HITL]
  |-- invoice  -> [Invoice Subgraph: extract line items -> validate -> ERP write]
  |-- report   -> [Report Subgraph: summarize -> index in vector store]
  +-- form     -> [Form Subgraph: extract fields -> validate -> database write]
        |
  [Aggregator: log results, route errors to dead-letter queue]
```

**Batch processing -- Map-Reduce:**
```
[Document Batch]
  |
fan_out -> [Process Doc 1] [Process Doc 2] ... [Process Doc N]  <- parallel
  |
fan_in  -> [Aggregator: collect results, report failures]
```

**When to use:** Invoice processing, contract analysis, form extraction, any pipeline with heterogeneous document types and strict output schemas.
**When NOT to use:** Single document type with simple extraction (a single agent with structured output suffices), documents requiring deep reasoning rather than extraction.

### Behavioral Rules

- **Subgraph-per-document-type.** Each document type gets its own compiled subgraph with its own state schema, extraction prompts, and validation logic. Share the parser subgraph but nothing else. This prevents 30-50% of redundant development and enables independent versioning. See `patterns.md` Section 3.5 for subgraph implementation.
- **Structured output extraction.** Use LLM-with-structured-output for all extraction steps. The structured schema acts as implicit validation -- if the LLM cannot fill the schema, extraction fails explicitly rather than silently.
- **Validation before write.** Every write to an external system is preceded by a deterministic validation node (no LLM). If validation fails, route to dead-letter queue with error context, not to a retry loop.

### State Shape

```python
class DocumentState(TypedDict):
    document_id: str
    raw_content: str
    parsed_text: str
    parsed_tables: list[dict]
    doc_type: str
    doc_type_confidence: float
    extracted_fields: dict                                         # schema varies by doc_type
    validation_errors: list[dict]                                  # {field, error}
    downstream_write_status: Literal["pending", "success", "failed", "review"]
    processing_stage: str
```

### Guardrails

| Guardrail | Rule |
|---|---|
| Dead-letter queue | All validation failures and extraction errors route to DLQ with full context. Never silently discard. |
| Idempotency | Every document write includes `document_id` as idempotency key. Reprocessing is safe. |
| Type classifier fallback | If `doc_type_confidence < 0.7`, route to human review before processing. See `structured-classification.md`. |
| Batch concurrency cap | Limit parallel document processing to avoid rate limit cascades. |
| PII handling | Scan extracted fields for PII before logging or storing. Redact before telemetry. |

### Failure Modes

| Failure | Root Cause | Fix |
|---|---|---|
| Wrong type classification | Ambiguous or mixed-type documents | Confidence threshold + human review path |
| Schema mismatch on write | Extracted field does not match target schema | Validation node between extraction and write -- always |
| Repeated processing | No idempotency on retries | Document ID as idempotency key on all writes |
| Parser failure on complex layouts | Tables, charts, multi-column text | Use VLM-powered parser for complex layouts |

## RAG / Knowledge Retrieval

### What Distinguishes It

Retrieval quality is the floor under which no generation quality improvement can compensate. A single retrieval call is not an agent. Agentic RAG replaces "embed-search-generate" with a cyclic decision loop: the agent plans, executes retrieval tools, evaluates results, and decides whether to re-retrieve or generate.

### Topology

**Corrective RAG (CRAG) Loop:**
```
[Query]
  |
[Hybrid Retrieval: dense + sparse + graph in parallel]
  |
[Relevance Evaluator: confidence score on top-k results]
  |-- confidence > HIGH   -> generate directly
  |-- confidence > LOW    -> augment with web search -> generate
  +-- confidence < LOW    -> rewrite query -> retry (max 3 cycles)
        |
  [Generator]
```

**Adaptive Router (cost-critical):**
```
[Query]
  |
[Complexity Classifier]
  |-- simple factoid  -> dense retrieval only (k=3)
  |-- moderate        -> hybrid retrieval (dense + sparse)
  +-- complex multi-hop -> full agentic CRAG loop
```

> **Design axiom: Tiered escalation.** Routing simple queries to dense-only retrieval cuts cost 40-60% without meaningful quality loss.

**Multi-Agent RAG (when single-agent context overflows):**
```
[Query]
  |
[Decomposer: breaks into sub-queries]
  |-- [Retriever Agent A: Domain 1]
  |-- [Retriever Agent B: Domain 2]
  +-- [Retriever Agent C: Web]
        |
  [Evaluator Agent: score and filter retrieved evidence]
  |
  [Generator]
```

**When to use:** Knowledge-intensive question answering, document search with quality requirements, any system where retrieval quality directly determines output quality.
**When NOT to use:** All required information is already in the context window, corpus is small enough for full inclusion, tasks requiring reasoning over structured data (use Data Analysis instead).

### Behavioral Rules

- **Hybrid retrieval is the production default.** Dense retrieval fails on exact numeric/identifier queries. Sparse (BM25/SPLADE) fails on semantic queries. Parallel dense + sparse, fused via Reciprocal Rank Fusion (RRF), handles both. RRF requires no labeled data for tuning.
- **Query rewriting on low confidence.** When retrieved documents are irrelevant, the problem is usually the query, not the corpus. Rewrite from a different angle before declaring failure. Cap at 3 rewrites.
- **Cache by embedding similarity.** Cosine distance < 0.05 between new query and prior query = reuse cached documents. Highest-ROI optimization for high-volume retrieval.

### State Shape

```python
class RAGState(TypedDict):
    query: str
    rewritten_queries: Annotated[list[str], operator.add]
    retrieved_docs: Annotated[list[dict], operator.add]            # {id, content, source, score, retrieval_method}
    confidence: float
    retry_count: int
    web_augmentation_used: bool
    generated_answer: str
    faithfulness_score: float                                      # post-generation check
    status: Literal["retrieving", "evaluating", "generating", "done", "failed"]
```

### Guardrails

| Guardrail | Rule |
|---|---|
| Retry budget | Max 3 retrieval cycles. After 3 failed attempts, return low-confidence answer with disclaimer. |
| Parallel retrieval | Fire dense + sparse retrievers concurrently. Sequential retrieval is a latency anti-pattern. |
| Cache by embedding similarity | Cosine distance < 0.05 = reuse cached retrieval result. |
| Post-generation faithfulness check | Score generated answer against retrieved context. Flag when faithfulness < 0.7. |
| Source attribution required | Every factual claim in the generated answer cites a retrieved document by ID. |

### Failure Modes

| Failure | Root Cause | Fix |
|---|---|---|
| Retrieval loop without convergence | Query rewriting cycling on same semantic space | Require each rewritten query to be substantively different; hash check |
| Hallucinated answer despite retrieval | Generator ignores retrieved context | Faithfulness scoring; prompt: "Answer only from provided context" |
| Embedding space collapse | All documents have high cosine similarity (> 0.8 mean pairwise) | Inspect embedding distribution; switch to better-calibrated embedding model |
| CRAG trigger rate too high | First-stage retrieval quality is poor | Invest in hybrid retrieval; improve chunking strategy |

## Autonomous Task Execution

### What Distinguishes It

The agent takes actions with real-world side effects: writing files, calling APIs, modifying databases, sending communications. Irreversibility is the defining constraint. Every design decision must account for the cost of a wrong action that cannot be undone.

### Topology

**Sequential + HITL gates:**
```
[Goal]
  |
[Planner: produce action plan + tag each action reversible/irreversible]
  |
[HITL gate: human reviews plan before any execution begins]
  |
[Executor: ReAct Loop]
  |-- call tool
  |-- observe result
  |-- update plan if needed
  +-- check if next action is irreversible -> HITL gate if yes
  |
[Verifier: check goal completion criteria]
  |
[HITL gate: human reviews output before delivery]
```

**When to use:** Workflow automation, DevOps agents, data pipeline management, any agent with write access to external systems.
**When NOT to use:** Read-only analysis tasks (no irreversible actions to gate), tasks where all actions are low-stakes and fully reversible.

### Behavioral Rules

- **Plan-and-Execute with irreversibility tagging.** The planner tags each action `reversible` or `irreversible`. Execution proceeds automatically for reversible actions. HITL gate fires before every irreversible action.
- **Minimal footprint principle.** Request only the permissions required for the current task. Prefer reversible over irreversible actions when both achieve the goal. Default to read-only; escalate to write access only when the plan requires it.
- **Confidence-gated execution.** Before executing a tool call, score the agent's confidence. Below threshold: interrupt for human confirmation rather than proceeding on a guess.

### State Shape

```python
class ExecutionState(TypedDict):
    goal: str
    action_plan: list[dict]                                        # {action_id, description, tool, args, reversible, status}
    execution_log: Annotated[list[dict], operator.add]             # {action_id, tool, args, result, timestamp} -- append-only, immutable
    current_action_index: int
    hitl_decisions: Annotated[list[dict], operator.add]            # {action_id, decision, timestamp}
    goal_completion_score: float
    status: Literal["planning", "awaiting-approval", "executing", "verifying", "done", "failed"]
```

**Execution log is append-only and immutable.** It is the audit trail. Never modify a prior log entry. This enables rollback reasoning: "What actions have been taken? Which can be reversed?"

### Guardrails

| Guardrail | Rule |
|---|---|
| Irreversible action gate | Never execute an irreversible action without explicit human approval in the current session. |
| Scope restriction | Tool permissions are scoped to the task. Enforce at the tool level, not the prompt level (`production.md` Tool Permission Scoping). |
| Rate limiting on external calls | Cap calls to external APIs per session. Prevents runaway loops from triggering downstream systems repeatedly. |
| Rollback plan | For every irreversible action, the planner must also generate a rollback procedure. If rollback is impossible, escalate. |
| Audit log integrity | Execution log is write-once. Any attempt to overwrite or delete a log entry is a failure condition. |

### Failure Modes

| Failure | Root Cause | Fix |
|---|---|---|
| Irreversible action on wrong assumption | Plan made on incomplete information | Validate key assumptions before irreversible actions; HITL gate with assumption summary |
| Tool call loop | Agent retries a failing tool indefinitely | Per-tool retry cap; error escalation path after N failures |
| Scope creep | Agent calls tools outside task scope | Enforce tool permission list at the framework level, not the prompt level |
| Prompt injection via tool result | External content in tool output hijacks agent behavior | Sanitize tool results; treat external content as data, not instructions (`production.md` MCP Security) |
